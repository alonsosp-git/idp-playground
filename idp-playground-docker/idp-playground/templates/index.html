What is the Token Manager?

     

      Every time a user authenticates through any of IDP-Playground's protocols (OIDC, SAML, OAuth2, WS-Fed),
      a signed JWT (JSON Web Token) is issued and recorded here.
      Tokens carry the user's identity claims (name, email, groups, department) and are valid for a configurable lifetime.


      View — inspect the full decoded token payload and claims.

      Revoke — immediately invalidate a token before it expires (e.g. user logged out or account suspended).

      Delete — permanently remove a revoked token record from the database.

      Delete All Revoked — clean up all revoked records at once.

● IDP-TS
Signing Keys
// JWK Set — RSA key management

    
   


   
   

    

     
What are Signing Keys?

     

      Signing Keys are the RSA-2048 keypairs IDP-Playground uses to digitally sign all JWT tokens.
      When a user logs in, their identity token is signed with the active key's private half.
      The receiving application verifies the signature using the public half — available at the JWKS endpoint.


      Certificate (Base64) / Certificate (Raw) — Export the signing cert to configure trust in your SAML Service Provider or WS-Fed Application.

      Download .pem — Raw public key for manual JWT verification.

      JWKS Endpoint — Standard URL your apps use to auto-discover the public key (OIDC / OAuth2).
     

    

   


   
   

    

     
This key is server-wide — not per user

     

      A signing key such as idp-key-001 is IdP infrastructure, shared by every user and every application. It is created once when IDP-Playground first starts — enabling MFA or issuing a certificate for a specific user does not create a key. That is why it carries no username: like a web server's TLS certificate, one key signs all tokens, and apps verify them with its public half via the JWKS endpoint. This is the same concept as the token-signing certificate in AD FS (Service → Certificates), the signing keys in Okta, and the token-signing certificate in Entra.


      The per-user objects are user certificates (Certificate-Based Auth), issued by the IDP-Playground Root CA under Security → IDP-Playground Root CA and assigned to a user in IDP-DS → Users → MFA. Those do identify an individual (their CN is the user). Signing keys sign tokens; user certificates identify people — two different things.

🪪 IDP-Playground Root CA

      
Certificate Authority for Certificate-Based Authentication

     

    

    
    

     

      The IDP-Playground Root CA is the Certificate Authority that signs all user client certificates.
      Think of it like an internal corporate CA — it vouches for the identity of each user's digital certificate.

      How it works:
      1. Generate the Root CA here (once).  
      2. Issue client certs to users via IDP-DS → Users → MFA → Certificate tab.  
      3. Users download the .p12 bundle and install it in their browser.  
      4. On the login page they can choose "Sign in with Certificate" instead of a password.
      
SSO Certificates (exported from each app's Metadata tab) are different —
      those are the IDP's signing certificates used by your Service Provider to verify token signatures.
      The Root CA here is for user identity certificates.

// Reference — Security Assertion Markup Language IDP metadata

The application (Service Provider) redirects the user to IDP-Playground (Identity Provider); after authentication IDP-Playground POSTs a signed SAML assertion to the app's ACS URL. ● IDP-TS
WS-Federation
// Reference — legacy Microsoft federation (SAML tokens over WS-Trust)

   
WS-Federation is a browser-based SSO protocol from the Microsoft / WS-* era. The application redirects the user to IDP-Playground's Security Token Service (STS); after authentication the STS POSTs a signed SAML 1.1 / 2.0 token back to the app, which validates the signature and starts a session. It works like SAML 2.0 and OIDC but uses different terminology and query parameters, and is common in legacy Microsoft stacks (SharePoint, older .NET, Dynamics). It is effectively in maintenance mode — support it where it exists, prefer OIDC / SAML for new apps. This page is reference information about the protocol.

   

    
Endpoints

    

     
STS / SignIn URL/wsfed

     
SignOut URL/wsfed?wa=wsignout1.0

     
Token typeSAML 1.1 / 2.0 assertion

     
Federation metadata/apps/<id>/metadata.xml

    

   

   

    
Request Parameters

     

      
wawsignin1.0 / wsignout1.0

      
wtrealmapp identifier (like client_id)

      
wreplyreturn URL for the token

      
wctxopaque state echoed back

      
wresultthe signed token POSTed back

     

    

    
How it compares

     

      App identifier → wtrealm (SAML: EntityID · OIDC: client_id)

      Return URL → wreply (SAML: ACS URL)

      Token → SAML assertion (OIDC: JSON id_token)

      Transport → browser POST / redirect

      Ecosystem → primarily Microsoft / legacy
